tentgent tui — local operator console

A daemon-first, keyboard-only terminal UI for the Tentgent local AI runtime. Two-pane layout, dense grids, explicit confirmation for destructive work, and the daemon connection state always visible in the header.

Wireframes are drawn at 120 × 36 (primary) with a 100 × 30 fallback variant in Appendix A. Implementation-neutral — fits ratatui / crossterm.

Information density

  • Five blocks — Runtime, Daemon, Auth, Inventory, Warnings — each capped at ~6 rows so the screen fits 36 lines without scroll.
  • Counts mirror the left-nav badges so a glance at either side gives the same answer.
  • Warnings deep-link by selecting the originating row in the relevant section on Enter.

Why daemon-down is its own variant

  • The Status screen is the only one that must work without the daemon. The other six show a single banner and bail.
  • Doctor expands to fill the freed space — diagnosis is the job here.
  • Inventory counts collapse to ? rather than guess from cache.

Reused shell

Mixed local + provider

  • Local servers (llama.cpp / Python runtime) and provider proxies (openai, anthropic) live in the same list — the operator thinks "where does the chat go", not "is this hosted".
  • Provider rows show no PID / port / vram; their detail surfaces upstream model id and last error instead.

Logs view

  • l swaps the right pane for a full-height log stream.
  • f toggles follow-tail · / filters · Esc back.

Confirmation pattern

  • Default focus is No. ↵ = No, y = Yes.
  • Double border distinguishes modals from panes; rest of screen dims.
  • Irreversible variants require typing the short ref to confirm.

Session chat

only top-bottom split — header strip · scroll · input

The one screen where the right pane stacks vertically: a header strip showing the session's bound server / adapter / token state, a scrollable bounded message history, and a sticky input at the bottom. Compaction and provider state surface as a single colored strip.

State strips

Replaces the metadata strip when the session enters a non-idle or error state. Single line — the layout never reflows.

compactiontokens 7 988 / 8 192 compaction: ◐ running… est 4s
session_busy! another request is in flight (started 14:09:11) [w] wait [k] kill
server✕ server_not_running — srv_local_qwen3_8b is stopped [s] start [v] choose another
compaction✕ compaction_failed — last attempt 14:07:55 [r] retry [o] open log

Why top-bottom here only

  • Chat needs full reading width. A vertical split would clip messages and force the user to wrap mid-thought.
  • The header strip preserves "what is this chat bound to" — it's the deepest screen and easy to lose context in.

Bounded history

  • The session-store contract bounds what's kept in memory; older messages collapse into a "32 earlier messages" indicator.
  • Compaction status is a first-class strip element — never a hidden background process.

Screen keys

↵send S-↵newline in input tswitch session vswitch server aswap adapter ccompact now xreset memory (confirm)

Action tray

  • Pinned at the bottom of the detail pane so the actions stay in eye-line with what they'd act on.
  • Mirrors the bottom hint bar but with full labels — a discoverability surface.

Provider gate

  • Synth and Eval always go through the modal — no exception, even with cached cost estimates.
  • d dry-run runs against 5 rows and returns cost + sample without spending the rest.

Artifact paths

  • The output filename uses an ISO timestamp so repeated runs never collide.
  • The path is shown before confirmation, so the operator can y yank it for a follow-up command.

Plans + runs in one view

  • Selecting a Plan filters the Runs list to runs of that plan.
  • Selecting a Run shows full live state in the detail pane — no further drill-down needed for a status check.

Live metrics

  • "Last" values refresh on each push from metrics.jsonl; no polling thrash.
  • M opens the full tabular metrics view with sparklines per series.

Screen keys

Read-mostly

  • Most settings are sourced from env / config — the source is annotated in parentheses so the operator knows where to change them outside Tentgent.
  • [edit] is the explicit signal that a field is mutable from inside the TUI.

.env guidance, not .env editor

  • The TUI does not edit .env in place — it opens it in $EDITOR via o.
  • R reloads the daemon after .env changes — explicit, never silent.

Token surface

  • Per-provider source is shown — keychain, env VAR, or not configured.
  • Add-token flow lives behind a; mutation of existing tokens is intentionally out-of-scope for v1.
