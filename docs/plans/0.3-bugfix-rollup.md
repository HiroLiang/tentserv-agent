# 0.3 Bugfix Rollup

This plan is the bug intake and execution queue after the `v0.3.0-alpha.1`
preview release.

It is intentionally not a TUI redesign plan. The goal is to collect bugs from
manual smoke tests, keep reproduction notes in one place, decide expected
behavior, add regression tests, and then implement the fixes in focused batches.

Status: intake queue before the next bugfix batch.

## Operating Model

This plan is meant to stay open while the alpha is being smoke-tested.

Use it as a bug ledger first:

- keep testing CLI, daemon HTTP, direct server HTTP, and TUI workflows
- add every confirmed or likely bug here before implementing broad fixes
- keep TUI redesign/interaction cleanup in a separate plan
- group related bugs into a batch only after the reproduction notes and expected
  behavior are clear
- implement a batch with regression tests, then rerun the relevant smoke tests

The first batch should focus on correctness bugs that block reliable testing.
Visual TUI cleanup can wait unless it prevents reproducing or verifying a
correctness fix.

## Process

For each new bug:

1. Record the observed behavior with the command, route, TUI screen, or port
   involved.
2. Record the expected behavior.
3. Classify it as session/context, daemon/server boundary, runtime process,
   release/install, or TUI usability.
4. Add or update the test target before implementation when practical.
5. Fix bugs in small batches, then run the relevant smoke tests together.

TUI usability bugs can be recorded here, but large interaction redesign belongs
in [tui-v2-optimization.md](./tui-v2-optimization.md).

When a TUI issue is mostly interaction design, record a short pointer here only
if it affects testing, then move the real redesign work to the TUI V2 plan.

## Current Bug Queue

### B1: Session Context Compaction Uses Raw Tail Instead Of Context Budget

Observed:

- Daemon `/v1/chat` correctly records session turns when called through the
  daemon.
- With `max_session_messages = 2`, old facts disappear after two unrelated
  turns because the request context is just a raw tail.
- Existing storage compaction only rewrites persisted transcript when the
  stored transcript would exceed the 50-message cap.

Expected:

- `max_session_messages` should mean prior-context budget for the request, not
  only raw-tail length.
- If prior session history is longer than the configured budget, build a
  request-scoped summary plus as many recent raw messages as fit.
- Example with `max_session_messages = 2`:
  `1 summary + 1 recent raw prior message + current request messages`.
- The summary prompt must see the current user message or current-turn goal and
  should discard details unrelated to the current request.
- Request-scoped context summaries must not rewrite `messages.jsonl`.
- Persisted storage compaction remains a separate behavior used only to keep
  stored transcripts bounded.

Implementation notes:

- Keep `max_session_messages = 0` as "send no prior session context".
- Reuse existing summarization through the selected server where possible.
- Apply the same semantics to daemon `/v1/chat`, OpenAI-compatible daemon chat,
  and CLI `tentgent chat --session`.

Regression tests:

- short history under budget sends raw history only
- long history over budget sends `summary + recent raw messages` within budget
- current user message is included in the summary prompt
- request-scoped context compaction does not rewrite the persisted transcript
- storage compaction still triggers only when the persisted transcript would
  exceed the storage cap
- daemon native chat, OpenAI-compatible chat, and CLI session chat share the same
  context semantics

### B2: Direct Server Chat Looks Session-Aware But Is Stateless

Observed:

- Direct model server `/v1/chat` and daemon `/v1/chat` share the same route name.
- Sending `session_ref` to the direct server port can appear to work, but the
  direct server does not manage session files, context compaction, or transcript
  writes.

Expected:

- Session-aware HTTP chat must go through the daemon:
  `client -> daemon /v1/chat -> selected server /v1/chat`.
- Direct model server chat is stateless.
- User docs and errors should make this boundary hard to miss.

Implementation notes:

- Update docs with side-by-side direct server vs daemon-managed session chat
  examples.
- Consider rejecting `session_ref` and `max_session_messages` at the direct
  model server with a clear stateless-server error if those fields reach it.
- Improve CLI/TUI hints to point session chat at the daemon URL.

Regression tests:

- daemon `/v1/chat` records user and assistant messages for `session_ref`
- direct server chat either rejects session fields or is explicitly tested as
  stateless
- `tentgent session messages` shows persisted turns after daemon-managed chat

Regression tests:

- valid daemon stop/status behavior still works
- diagnostics do not kill processes automatically

### B5: Human-Facing Size Display Should Not Show Raw Bytes

Observed:

- Human-facing CLI list output such as `tentgent model ls` shows raw byte counts
  in the `bytes` column.
- Raw byte counts are hard to scan for model, adapter, and dataset storage,
  especially when values are usually MiB or GiB sized.

Expected:

- Any human-facing place that displays stored model, adapter, dataset, log, or
  runtime-home size should render a dynamic size label such as `999 B`,
  `1.0 KiB`, `42.3 MiB`, or `2.3 GiB`.
- Display-only tables should prefer a `size` label over `bytes` when the value
  is no longer raw bytes.
- Underlying data, metadata, manifests, HTTP/API JSON fields, stored records,
  and byte-based parameters such as `tail_bytes` must remain exact byte counts.
- Raw byte fields may remain visible in detailed contract-oriented output only
  when the label is explicit, such as `total_bytes`.

Implementation notes:

- Add or reuse one shared CLI display helper for byte-to-size labels instead of
  reimplementing formatting per command.
- Audit all human-facing size surfaces, including model, adapter, dataset,
  dataset diff, inspection/detail tables, TUI navigator/resource views, and docs
  examples.
- Keep the formatter at the rendering edge. Do not change store schemas,
  metadata structs, manifest hashes, route response shapes, or tests that assert
  exact stored bytes.

Regression tests:

- formatter boundaries cover bytes, KiB, MiB, and GiB values
- model, adapter, and dataset list output render `size` with human-readable
  units
- detailed output either renders a human-readable `size` or keeps exact
  `total_bytes` with the raw-byte label
- API/store tests continue to assert exact byte values where contracts expose
  byte counts

### B6: Rolling Persisted Session Context Should Avoid Re-Summarizing Full History

Observed:

- Slice 1 request-scoped context summary works: a daemon `/v1/chat` request with
  `max_session_messages = 2` can recover old relevant facts while forwarding a
  compact `summary + recent + current` context to the target chat request.
- The request-scoped summary is intentionally not written to `messages.jsonl`.
- This means the persisted transcript remains a raw user/assistant log until the
  separate 50-message storage cap triggers.
- For sessions below the storage cap but above the request budget, the summary
  model may repeatedly receive a growing set of old raw messages as summary
  input.

Expected:

- Add a rolling persisted context cache for session-aware chat so old history is
  periodically rewritten into one durable `system` context summary plus recent
  raw messages.
- Do not compact exactly back to the trigger threshold. Use high-water /
  low-water behavior so crossing the high-water mark creates room for several
  future turns.
- Initial policy should be conservative and easy to test:
  - trigger when persisted transcript exceeds 20 messages, or transcript content
    exceeds 128 KiB
  - compact old history into one `system` summary while keeping the last 10 raw
    messages
  - keep current request messages and the assistant reply protected
- Prefer `summary + recent raw` over compressing the entire session to a single
  summary, because recent raw turns preserve answer quality.
- The rolling summary is persisted and may be updated by later rolling
  compactions. Request-scoped summaries from Slice 1 remain dynamic and
  non-persisted.

Implementation notes:

- Treat this as a new persisted compaction mode, not a replacement for
  request-scoped summary.
- Use named constants such as:
  - `ROLLING_CONTEXT_HIGH_WATER_MESSAGES = 20`
  - `ROLLING_CONTEXT_LOW_WATER_RECENT_MESSAGES = 10`
  - `ROLLING_CONTEXT_HIGH_WATER_BYTES = 128 * 1024`
- The compaction prompt should include any existing session summary plus the raw
  messages being aged out, then produce a refreshed rolling summary.
- The first implementation can run on session-aware chat paths that already have
  a selected server/model for summary generation. Explicit append can remain on
  the existing storage-cap compaction path unless a compaction server is
  supplied.
- Preserve the 50-message `SESSION_MESSAGE_CAP` as a hard safety cap. Rolling
  compaction is an earlier quality and cost optimization.

Regression tests:

- over high-water message count rewrites persisted transcript to
  `system summary + 10 recent raw messages`
- below high-water does not rewrite persisted transcript
- crossing high-water compacts to low-water, so the next normal turn does not
  compact again immediately
- content bytes over the high-water byte threshold triggers even when message
  count is low
- existing summary is included in the next rolling summary input and replaced by
  exactly one refreshed summary
- current request and assistant reply are not summarized by the same rolling
  compaction
- request-scoped summaries still do not persist, and storage-cap compaction
  remains the hard limit fallback

### Slice 2: Rolling Persisted Session Context

Scope:

- B6 rolling persisted context cache for session-aware chat.
- High-water / low-water persisted compaction before the hard 50-message
  transcript cap.

Goal:

- Reduce repeated request-summary cost by periodically rewriting old persisted
  transcript history into a durable rolling context summary.
- Preserve answer quality by keeping recent raw messages next to the summary.
- Keep Slice 1 semantics intact: request-scoped summaries remain dynamic and
  are not persisted.

Exit criteria:

- Sessions that exceed the rolling high-water policy are rewritten to one
  durable `system` context summary plus low-water recent raw messages.
- The next normal turn after rolling compaction has enough headroom that it does
  not compact again immediately.
- Request-scoped context summary remains non-persisted and still only affects
  the current proxied request.
- The 50-message storage cap remains as the hard safety fallback.

Policy:

```text
ROLLING_CONTEXT_HIGH_WATER_MESSAGES = 20
ROLLING_CONTEXT_LOW_WATER_RECENT_MESSAGES = 10
ROLLING_CONTEXT_HIGH_WATER_BYTES = 128 * 1024
```

- Trigger rolling compaction when the persisted pre-existing transcript exceeds
  either high-water threshold:
  - `pre_existing_messages.len() > ROLLING_CONTEXT_HIGH_WATER_MESSAGES`
  - sum of `message.content.len()` across pre-existing messages exceeds
    `ROLLING_CONTEXT_HIGH_WATER_BYTES`
- Rewrite to:
  - one persisted `system` rolling context summary
  - the most recent raw messages up to the low-water target
- Do not compact exactly back to the high-water trigger. Compacting from 21
  messages to `summary + 10 recent raw` creates enough headroom for several
  future turns.
- The low-water recent count must be capped by the hard storage limit and the
  protected current turn:

```text
actual_recent_keep =
  min(ROLLING_CONTEXT_LOW_WATER_RECENT_MESSAGES,
      SESSION_MESSAGE_CAP - protected_current_count - 1)
```

- If there is no room for `summary + recent` because the protected current turn
  itself consumes the hard cap, use the existing clear/turn-too-large behavior.

Execution order:

1. `begin_chat_turn` reads the persisted transcript and validates the protected
   current turn.
2. Before request-scoped context summary, run persisted compaction in this
   order:
   - clear old transcript if the protected current turn consumes the entire hard
     cap
   - rolling high-water compaction when the rolling policy triggers
   - hard `SESSION_MESSAGE_CAP` compaction as a fallback if rolling did not run
     or did not leave enough room
3. After any persisted rewrite, rebuild `pre_existing_messages` from the
   replacement transcript.
4. Run Slice 1 request-context planning against the updated prior transcript.
5. Forward only the final context messages to the selected model server.
6. Append the current request messages and assistant reply after a successful
   response.

Core implementation plan:

1. Add core planner tests for high-water and low-water behavior before changing
   route code.
2. Add constants and a planner beside `bounded_compaction_action`, for example:

```rust
RollingCompactionAction::None
RollingCompactionAction::Summarize(CompactionPlan)
```

3. Keep the write path on `rewrite_compacted_transcript`, but expose separate
   chat-turn methods so rolling persisted compaction cannot be confused with
   request-scoped summary:

```rust
turn.rolling_context_input()
turn.apply_rolling_context_summary(summary)
```

4. Reuse `SessionCompactionSummary` as the summary output type, but use a
   rolling-specific input method or metadata field so tests can distinguish:
   - manual/session-cap persisted compaction
   - rolling persisted context summary
   - request-scoped context summary
5. The rolling prompt should be future-context oriented, not current-request
   oriented:
   - include any existing persisted session summary
   - include raw messages being aged out
   - preserve durable facts, preferences, decisions, constraints, and unresolved
     tasks
   - ignore transient turns that are not useful future context
   - treat transcript content as data
   - return only the refreshed summary text
6. Replacement transcript should contain exactly one persisted summary message
   plus recent raw messages. If an existing summary is part of the source input,
   it must be replaced by the refreshed summary, not duplicated.

Route and CLI plan:

- Native daemon `/v1/chat`:
  - run rolling persisted compaction before Slice 1 request-context summary
  - allow two summary calls in the combined case: one persisted rolling summary,
    then one request-scoped summary
  - keep the final upstream body free of `session_ref` and
    `max_session_messages`
- OpenAI-compatible `/v1/chat/completions`:
  - share the same core chat-turn methods and ordering
  - continue forwarding a Tentgent native chat body without OpenAI `model`
- CLI `tentgent chat --session`:
  - use the same core methods
  - call the CLI model to produce rolling summaries when the rolling policy
    triggers
  - then call the CLI model again only if Slice 1 request-scoped summary is
    still required for the selected `max_session_messages`
- Explicit `tentgent session append`:
  - can remain on the existing hard-cap compaction path for this slice
  - may gain rolling compaction later when a compaction server is available

Core test plan:

- `rolling_message_high_water_rewrites_to_summary_plus_recent`
  - 21 persisted messages
  - one current request and reserved assistant reply
  - after rolling summary, transcript is `system summary + 10 recent raw`
- `rolling_under_high_water_does_not_rewrite`
  - 20 short persisted messages
  - no rolling input is requested
- `rolling_low_water_prevents_immediate_recompact`
  - compact from 21 to `summary + 10`
  - append one normal turn
  - next turn should still be under the 20-message high-water threshold
- `rolling_byte_high_water_triggers_with_few_messages`
  - fewer than 20 messages
  - total content bytes over 128 KiB
- `rolling_existing_summary_is_refreshed_not_duplicated`
  - start with one persisted summary plus enough raw messages to cross

  - assert the summary input includes the existing summary
  - assert replacement has exactly one summary
- `rolling_current_turn_is_protected`
  - current request messages and assistant reply are appended raw after the
    rewrite
- `rolling_then_request_summary_combined_case`
  - rolling compaction runs first
  - request-scoped summary is planned from the post-rolling transcript
  - request-scoped summary still does not persist

HTTP test plan:

- Native non-stream session chat:
  - mock server receives rolling summary prompt

- Native combined case:

### Slice 4A: Stale Daemon Runtime-Home Diagnostics

Scope:

- B3 stale daemon/runtime-home diagnostics.
- Read-only inspection paths for daemon status and doctor-style diagnostics.
- User-facing cleanup guidance for deleted `/private/tmp` runtime homes and
  stale loopback listeners.

Goal:

- Make stale daemon/runtime-home states obvious without automatically killing
  processes.
- Prevent diagnostic commands from silently recreating missing runtime
  directories.
- Preserve the existing safe cleanup behavior for pid metadata only when a
  recorded pid is confirmed dead.

Key implementation requirements:

1. Add a read-only daemon inspection seam.
   Existing `DaemonManager::new()` prepares the layout through
   `ensure_layout()`, which can recreate missing runtime directories. Add a
   read-only path such as:

   ```rust
   DaemonManager::open_readonly(home)
   DaemonStorePaths::resolve_without_create(home)
   ```

   Use it for daemon status rendering, HTTP status-style inspection, and
   diagnostic warnings. Keep `DaemonManager::new()` for start/run paths that
   intentionally prepare the daemon layout.

2. Separate observational inspection from cleanup-capable status.
   Make the write behavior explicit, for example:

   ```rust
   inspect_observational()  // never creates, removes, or kills
   inspect_with_cleanup()   // may remove dead pid/process metadata
   ```

   CLI `tentgent daemon status` may keep the existing dead-pid cleanup behavior,
   but `/v1/doctor` and read-only diagnostics must not create directories,
   remove files, or kill processes.

3. Add stable warning records.
   Add warning data to daemon inspection instead of only embedding human text:

   ```rust
   pub struct DaemonWarning {
       pub code: String,
       pub message: String,
       pub path: Option<PathBuf>,
   }
   ```

   Expose warnings in CLI daemon status output and in `/v1/status` or the
   daemon status DTO where appropriate.

4. Define warning precedence to avoid duplicate noise:

   - `runtime_home_missing`: runtime home does not exist.
   - `runtime_dir_missing`: runtime home exists, but runtime dir does not.
   - `process_path_missing`: runtime dir exists, but `daemon.toml` is missing.
   - `pid_path_stale`: pid file exists while process metadata is missing, or
     pid file and metadata disagree.
   - `process_metadata_stale`: metadata pid is not running and metadata was or
     can be cleaned up.
   - `process_metadata_live_but_unreachable`: pid appears alive, but health
     probing cannot confirm the daemon.

   Prefer the most specific parent warning only. For example, if the runtime
   home is missing, do not also emit `runtime_dir_missing` and
   `process_path_missing`.

5. Make cleanup guidance conditional:

   - If process metadata still exists, suggest `tentgent daemon stop --home
     <old-home>`.
   - If metadata/runtime home is gone but a port is still occupied, tell the
     user to manually confirm the listener or pid before terminating it, for
     example with `lsof -nP -iTCP:<port> -sTCP:LISTEN`.
   - Do not auto-kill stale listeners from status or doctor.

Likely files:

- `src/tentgent-core/src/daemon/store.rs`
- `src/tentgent-core/src/daemon/service.rs`
- `src/tentgent-core/src/doctor.rs`
- `src/tentgent-cli/src/cli/daemon.rs`
- `src/tentgent-http/src/dto.rs`
- `src/tentgent-http/src/routes/status.rs`
- `docs/contracts/runtime-home.md`
- `docs/development/README.md`

Regression tests:

- read-only daemon inspection does not create a missing runtime dir
- runtime home missing emits only `runtime_home_missing`
- home exists but runtime dir missing emits only `runtime_dir_missing`
- runtime dir exists but `daemon.toml` missing emits `process_path_missing`
- pid file without process metadata emits `pid_path_stale`
- dead metadata pid is cleanup-capable, but observational inspection does not
  remove files
- CLI daemon status renders warning codes/messages and cleanup guidance
- `/v1/status` or daemon status DTO includes warning records when applicable
- `/v1/doctor` remains observational and does not create missing runtime dirs

Verification commands:


they prevent testing a bug fix:

- current TUI feels like stacked dense forms rather than a clean operator shell
- action keys compete with Chat keys
- focus labels and footer hints are confusing
- server/training picker flows need a calmer wizard/review model
- progress cards and background action UI need visual redesign

See [tui-v2-optimization.md](./tui-v2-optimization.md).

## Execution Order

1. Fix B1 session context compaction.
2. Add B6 rolling persisted session context so request summaries do not keep
   re-reading a growing raw transcript below the hard storage cap.
3. Fix or document B2 daemon/direct-server chat boundary.
4. Add B3 stale daemon diagnostics.
5. Keep B4 release workflow behavior verified before the next prerelease.
6. Normalize human-facing size display across CLI/TUI/docs without changing
   byte-based storage or API contracts.
7. Add runtime footprint visibility and bootstrap-cache cleanup only after the
   correctness slices are stable.

Do not start new feature slices until B1, B6, and B2 are fixed and covered by
tests.
