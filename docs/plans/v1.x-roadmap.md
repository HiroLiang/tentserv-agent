# v1.x Roadmap

Status: active long-term product roadmap after the `v1.0.0` stability release.

This roadmap tracks post-1.0 product evolution. It is for new capabilities,
architecture direction, and version-sized feature slices. Bug fixes, release
follow-up, and documentation cleanup belong in
[bugfix-maintenance-plan.md](./bugfix-maintenance-plan.md).

## Purpose

`v1.0.0` froze the documented stable compatibility promise. The `v1.x` line
should now add capabilities without weakening that promise:

- keep stable CLI and daemon REST behavior predictable
- keep provider-compatible support explicit and partial where documented
- make model, adapter, backend, runtime profile, and platform compatibility
  explainable before routing grows more dynamic
- add multi-model serving only when each configured route can be validated and
  inspected

## Roadmap Principles

- Keep the product surface CLI plus daemon REST.
- Prefer vertical feature slices that carry one operator-visible capability
  through contracts, domain behavior, entry points, docs, and tests.
- Keep native Tentgent request intent separate from provider-shaped payloads.
- Treat static model and adapter metadata as hints, not proof.
- Treat local verification on this machine as authoritative. Shared or built-in
  records can provide hints, but they must not replace local evidence.
- Avoid hidden fallback to another model, backend, provider, or generated
  metadata when a configured capability is missing.
- Keep large artifacts in filesystem stores; add indexed metadata only when it
  improves compatibility, proof, routing, or query behavior.

## v1.1 Candidate Scope

`v1.1.0` should focus on the smallest useful multi-model serving foundation:

- define a serving target, also called a cluster target, as one
  server-startable routing rule set
- validate configured `chat`, `embedding`, and `rerank` routes against known
  local model capability and runtime profile state
- add read-only serving target inspection before broad lifecycle automation
- start with native local routing across `chat`, `embedding`, and `rerank`
- add only the resource gates needed to avoid obvious multi-model conflicts
- keep unsupported provider-compatible and multimodal requests on clear errors

The first cluster slice should not implement automatic multimodal context
assembly, provider tool orchestration, OpenShell broker routing, shared
registries, or conversion automation.

The cluster candidate set is tracked in the focused
[cluster-serving-target-roadmap.md](./cluster-serving-target-roadmap.md)
sub-roadmap. Discuss and select items there before promoting them into
individual `v1.1.0` issues.

## Compatibility Foundations

The cluster direction depends on compatibility work that may be implemented as
part of `v1.1.0` or split into closely related slices:

- durable local compatibility proof storage for model, capability, backend,
  runtime package, platform/device class, adapter, and input/output shape
- tuple-aware model and LoRA compatibility gates so one successful adapter or
  model launch is not trusted for unrelated tuples
- actionable inspect and doctor output for verified, failed, stale, unknown, and
  unsupported local tuples
- bounded dynamic runtime routing where candidate attempts are documented,
  cached, and inspectable
- resource coordination for model mutation, adapter writes, conversion, cache
  updates, warm server ownership, and GPU/CPU/memory-sensitive jobs

feature release, or the maintenance plan.

| Bugfix And Maintenance | Patch fixes, stale docs, diagnostics polish, release follow-up, and repository hygiene for already released behavior. | Not part of the feature roadmap. | Compatibility State System | Route readiness, durable proof concepts, tuple-aware model/LoRA gates, support status, runtime profile visibility, and runtime failure evidence. | Define with cluster in `v1.1.0`; implement only the minimum needed for cluster slices. | SQLite or an equivalent indexed metadata layer is the likely long-term backend, but migration should happen incrementally by state family instead of as one broad storage rewrite. |
| Runtime Resource System | Delete protection, active route ownership, cancellation, shutdown, cleanup, and minimal resource gates for multi-model local serving. | Minimal cluster-required subset in `v1.1.0`; broader scheduler is not in `v1.1.0`. | Keep early work limited to protecting bound resources and active target routes. | Provider Orchestration System | Provider tool/function-call mapping, provider-specific extensions, and cloud rerank adoption. | Not in `v1.1.0`. | Keep provider-shaped behavior explicit at ingress/egress; do not pass through unsupported provider fields silently. |
| External Trust And Conversion System | OpenShell broker auth, shared compatibility registry, explicit model conversion, and generated metadata provenance. | Not in `v1.1.0`. | Local verification remains authoritative; conversion must be an explicit user action, not an execution-time patch. |

## Later 1.x Work

These directions are intentionally kept out of the first `v1.1.0` cluster MVP
unless a later planning pass narrows them into a safe vertical slice:

- automatic multimodal context assembly for files, images, audio, and video
- provider-compatible tool and function-call orchestration
- cloud rerank as an explicit provider or adapter family
- OpenShell-managed provider gateway or broker auth boundaries
- shared compatibility registry publish/consume behavior
- explicit conversion and metadata-generation workflows with provenance
- broader direct media server wrappers and typed streaming contracts beyond the
  already stable route families

## Issue Planning Rules

- Put feature and architecture issues here when they change product capability
  or public workflow shape.
- Put patch fixes, stale docs, release follow-up, and diagnostics polish in
  [bugfix-maintenance-plan.md](./bugfix-maintenance-plan.md).
- Use milestones for release targeting, such as `v1.1.0` or `v1.2.0`.
- Use the GitHub Project board for status, priority, area, and work type.
- Archive or rewrite this plan when its main tracks are completed or
  superseded.
