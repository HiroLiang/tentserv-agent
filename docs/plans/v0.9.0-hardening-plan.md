# v0.9.0 Hardening Plan

Status: active planning branch.

This plan breaks the `v0.9.0` hardening track into contributor-friendly
vertical issues. Each issue should carry the needed contract, implementation,
tests, and documentation for one operator-visible outcome. Do not split this
track into horizontal domain/store/CLI/docs/test issues unless one layer is an
independent design risk.

Parent tracking issue: `#16 v0.9.0 1.0 Hardening tracking`.

## Goal

`v0.9.0` should make the existing `v0.6.0` through `v0.8.0` surfaces reliable
enough to freeze in `v1.0.0`. It is not a feature expansion release.

The release should harden:

- stable and experimental API classification;
- provider and native conformance smoke coverage;
- install, upgrade, runtime bootstrap, doctor, and auth recovery;
- runtime cancellation, cleanup, and shutdown behavior;
- support proof retry and stale-state recovery;
- user-facing `1.0.0` readiness documentation.

## Completed Foundations

These tracks are already complete enough to treat as foundations rather than
reopen them during `v0.9.0` unless a bug is found:

- `v0.6.0`: provider compatibility matrix, examples, stable provider
  unsupported error codes, and OpenAI/Claude/Gemini compatibility fixtures.
- `v0.7.0`: support status vocabulary, built-in model support catalog, compact
  model/server support inspection, and doctor support visibility.
- `v0.8.0`: local chat and embedding runtime profiles, local server-start
  support gates, launch proof recording, proof clear/retry command, and
  configurable provider auth source modes.

`v0.8.0` auth source modes cover normal provider secret resolution through
`auto`, `keychain`, `file`, `env`, and `none`. Future OpenShell broker or
cluster-specific auth routing remains post-`1.0.0`.

## Out Of Scope

These items should not block `v0.9.0` or `v1.0.0`:

- multi-model serving target or cluster server launches;
- automatic local multimodal context assembly;
- provider-compatible tool-use orchestration;
- cloud rerank adapter support;
- advanced dynamic runtime routing across multiple candidate backends;
- shared/community support registry services.

The contracts should preserve room for those items, but the implementation
belongs after `v1.0.0`.

## Sub-Issues

### 1. Audit stable and experimental API surfaces before 1.0

Suggested branch: `feature/0.9-api-surface-audit`

Description:

```md
Audit Tentgent native APIs, provider-compatible APIs, CLI commands, response
shapes, and error shapes before the 1.0.0 freeze.

Scope:
- Classify each public surface as stable, experimental, internal, or deprecated.
- Identify fields, endpoints, and command names that must be renamed, frozen,
  or explicitly marked experimental before 1.0.0.
- Verify provider-compatible docs match actual daemon, local model server, and
  direct cloud server behavior.
- Update contract and user docs where wording implies stronger stability than
  the implementation provides.

Acceptance:
- A clear stable/experimental surface list exists.
- No documented stable endpoint relies on vague or untested behavior.
- Known experimental surfaces are marked clearly before 1.0.0.
```

### 2. Add 1.0 compatibility conformance smoke coverage

Suggested branch: `feature/0.9-conformance-smoke`

Description:

```md
Add a focused conformance smoke suite for supported native and
provider-compatible API surfaces before 1.0.0.

Scope:
- Cover supported OpenAI, Claude/Anthropic, and Gemini-compatible endpoint
  shapes.
- Cover native Tentgent chat, embeddings, rerank, and durable media job routes.
- Assert stable unsupported-provider error codes for known unsupported fields,
  content parts, operations, and capabilities.
- Keep tests offline and fixture-based unless explicitly marked as live smoke
  tests.

Acceptance:
- Supported compatibility routes have request/response regression coverage.
- Unsupported provider behavior remains stable and testable.
- Native and provider-compatible route contracts can be checked before release.
```

### 3. Harden runtime bootstrap, doctor, and recovery diagnostics

Suggested branch: `feature/0.9-runtime-doctor-recovery`

Description:

```md
Improve operator recovery for install, runtime bootstrap, auth, support proof,
and runtime profile problems.

Scope:
- Ensure `tentgent doctor` detects missing runtime profiles, missing provider
  auth, failed/stale support proofs, and common bootstrap problems.
- Keep summary output compact while moving long explanations and copyable
  commands into detail sections.
- Ensure runtime bootstrap failures point to a specific recovery command.
- Verify README and user docs route users to install, runtime, auth, support
  status, and doctor guidance.

Acceptance:
- Common runtime and auth failures have actionable next steps.
- Doctor output remains readable in narrow terminals.
- Users can recover from failed bootstrap or support proof states without
  reading implementation details.
```

### 4. Harden install and upgrade release paths

Suggested branch: `feature/0.9-install-upgrade-hardening`

Description:

```md
Harden direct installer, Homebrew, pinned-version install, and upgrade
documentation before 1.0.0.

Scope:
- Verify README, install docs, and version docs describe current install and
  upgrade flows.
- Confirm Homebrew upgrade guidance matches the tap release process.
- Confirm direct installer guidance points users to `tentgent doctor` after
  install.
- Add release-readiness checks for version metadata, release scripts, and
  Homebrew formula update workflow where missing.

Acceptance:
- Users can install, upgrade, verify, and recover using documented commands.
- Release docs match the actual tag and Homebrew workflow.
- Release preparation has repeatable checks before tagging.
```

### 5. Harden runtime cancellation, cleanup, and shutdown behavior

Suggested branch: `feature/0.9-runtime-cleanup-shutdown`

Description:

```md
Define and harden cancellation, cleanup, and shutdown behavior for long-running
daemon jobs, local server runtimes, and job workspaces.

Scope:
- Audit daemon-managed long-running jobs for cancellation and cleanup gaps.
- Define which cleanup behavior is guaranteed and which is best-effort.
- Ensure stopped or failed jobs do not leave confusing reusable state.
- Verify local server/runtime shutdown behavior is documented and tested where
  possible.
- Add recovery guidance for stale runtime or job workspace state.

Acceptance:
- Cancellation and cleanup semantics are documented.
- Runtime and job shutdown behavior is predictable enough for 1.0.0.
- Common stale workspace/runtime states have recovery guidance.
```

### 6. Harden support proof retry and stale-state recovery

Suggested branch: `feature/0.9-proof-retry-hardening`

Description:

```md
Harden support proof recovery flows for failed, stale, unknown, supported, and
verified model capability states.

Scope:
- Verify failed proof clearing and retry flows work consistently from CLI and
  documented APIs.
- Ensure stale proof states explain why old evidence no longer applies.
- Ensure `model inspect`, `server inspect`, and `doctor` describe support
  status consistently.
- Confirm server-start proof records include enough runtime profile and backend
  context to debug failures.

Acceptance:
- Failed and stale proof states are recoverable.
- Inspect and doctor output agree on support status and next actions.
- Proof records do not imply that a model is permanently broken when only one
  tuple failed.
```

### 7. Prepare 1.0 readiness documentation

Suggested branch: `docs/0.9-1.0-readiness`

Description:

```md
Prepare user-facing and contributor-facing 1.0 readiness documentation.

Scope:
- Add v0.9.0 version notes.
- Add a 1.0 readiness checklist.
- Document stable versus experimental surfaces.
- Keep localized README files as routing summaries only.
- Confirm README routes users to install, upgrade, commands, runtime, auth,
  provider compatibility, support status, and developer contracts.

Acceptance:
- Users can understand what 1.0.0 will promise.
- Contributors can see which surfaces are stable, experimental, or post-1.0.
- Documentation does not duplicate contract details unnecessarily.
```

### 8. Close v0.9.0 hardening release

Suggested branch: `release-0.9.0`

Description:

```md
Close the v0.9.0 hardening track and prepare the release branch.

Scope:
- Verify all v0.9.0 hardening issues are complete.
- Update version notes and README release notes.
- Run release-readiness checks.
- Confirm no stale contract wording remains.
- Prepare the v0.9.0 release PR and tag checklist.

Acceptance:
- v0.9.0 can be tagged after merge.
- Release notes clearly describe hardening scope.
- Remaining 1.0.0 and post-1.0 work is explicitly tracked.
```

## Execution Order

1. API surface audit.
2. Compatibility conformance smoke coverage.
3. Runtime bootstrap, doctor, and recovery diagnostics.
4. Install and upgrade release paths.
5. Runtime cancellation, cleanup, and shutdown behavior.
6. Support proof retry and stale-state recovery.
7. `1.0.0` readiness documentation.
8. `v0.9.0` release closeout.
