## Findings

| Finding | Impact | Owner |
| --- | --- | --- |
| The daemon contract previously described the entire HTTP daemon boundary as stable, while several route families still have `v0.9.0` hardening work. | User docs could imply stronger `1.0.0` guarantees for doctor output, shutdown cleanup, job cancellation, media jobs, LoRA runs, and support-proof retry than the implementation has earned. | #75 narrows wording; #77, #79, #80, and #81 finish the relevant behavior and readiness text. |
| The Python model runtime mounts execution routes under `/v1/*`, even though those routes are Rust-to-Python internals. | External callers could mistake runtime execution routes for public API routes if they see the process directly. | #75 marks the Python runtime surface internal; future runtime alias work should remain separate from this audit. |
| Local model-bound server uses a fallback proxy after mounting explicit provider-shaped ingress routes. | The fallback is useful for native routes, but docs should not imply every Python runtime route is public through the local server. | #75 documents fallback behavior as internal except for listed native routes. |
| Provider compatibility rows use `Supported`, `Partial`, and `Unsupported`, while the audit uses stability tiers. | Readers could confuse partial provider support with an unstable route. | #75 clarifies that partial provider coverage can still have stable route names and stable unsupported-error codes. #76 should add conformance smoke coverage for those stable errors. |
| Plural audio transcription routes `/v1/audio/transcriptions/jobs` coexist with the preferred singular upload route `/v1/audio/transcriptions/job`. | New users may see two similar routes without a clear preferred path. | #75 classifies the plural route as a compatibility alias. #81 can decide whether readiness docs should mention only the preferred route. |
| `PATCH /v1/models/{reference}` remains a legacy compatibility alias for updating one capability. | The alias should stay callable, but new clients should use the capability collection route. | #75 marks it deprecated. #81 should keep readiness docs focused on `POST /v1/models/{reference}/capabilities`. |
| LoRA training plan identity is contracted, but run execution and stale-process recovery are still being hardened. | Treating the full training run surface as stable before cleanup/recovery work would overstate `1.0.0` readiness. | #75 marks LoRA run surfaces experimental. #79 and #81 should revisit after cleanup/readiness work. |
