Closeout summary:

`v1.0.0` is not a feature expansion release. The release should prove the
documented stable Tentgent surfaces, keep unsupported provider-compatible
behavior predictable, and clearly route everything else to experimental or
post-1.0 work.

## Goal

`v1.0.0` should be cut only after users and contributors can answer:

- which CLI commands, daemon routes, local server routes, direct cloud server
  routes, and error shapes are stable;
- which surfaces remain experimental, internal, or deprecated;
- how provider-compatible partial support fails when a field, operation,
  endpoint family, or capability is unsupported;
- how model support status, runtime profile selection, and proof state can be
  inspected before relying on a stable workflow;
- how to install, upgrade, bootstrap the runtime, diagnose failures, and verify
  the published release artifacts.

## Foundations

The following tracks are complete enough to treat as foundations unless a
release-blocking mismatch is found:

- `v0.6.0`: provider compatibility matrix, provider examples, stable
  unsupported-provider error codes, and compatibility fixtures.
- `v0.7.0`: support status vocabulary, built-in support catalog, model/server
  support inspection, and doctor support visibility.
- `v0.8.0`: runtime profile planning, server-start support gates, launch proof
  recording, proof clear/retry command, and provider auth source modes.
- `v0.9.0`: API surface audit, conformance smoke coverage, install/runtime and
  doctor recovery, cleanup/shutdown semantics, proof retry/stale recovery, and
  readiness documentation.

## Pull Into 1.0

This release should finish and freeze:

- documented stable native and provider-compatible API surfaces;
- stable unsupported-field, unsupported-operation, unsupported-endpoint, and
  unsupported-capability errors;
- model support status, runtime profile, and proof-state inspectability for
  stable workflows;
- release trust checks for install, upgrade, Homebrew, doctor, runtime
  bootstrap, and GitHub Release assets;
- user-facing release notes and readiness routing that match tested behavior.

## Keep Post-1.0

These items should not block `v1.0.0`:

- SQLite-backed or shared/community proof registry;
- dynamic runtime routing across multiple candidate backends;
- grouped serving targets or cluster target configuration;
- automatic multimodal context assembly;
- OpenShell broker/gateway auth integration;
- provider-compatible tool-use orchestration;
- cloud rerank provider adapters;
- multi-model resource scheduling;
- automatic conversion or metadata synthesis workflows;
- broader direct media server wrappers or realtime media streaming contracts.

Current contracts should preserve room for those directions, but implementation
belongs after `v1.0.0`.
