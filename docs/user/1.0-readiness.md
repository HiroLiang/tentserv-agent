# 1.0 Readiness

This document is the user-facing and contributor-facing checklist for the path
to `1.0.0`. It explains what must be understandable before the `1.0.0` release.
It does not introduce new CLI commands, REST routes, runtime behavior, or schema
rules.

`1.0.0` is a stability promise for documented surfaces. It is not a feature
expansion milestone, and it does not mean every provider feature, every local
backend, or every model family has full parity.

The source of truth for public surface classification is
[api-surface-stability.md](../contracts/api-surface-stability.md). Use that
contract for the exact `stable`, `experimental`, `internal`, and `deprecated`
inventory.

## User Readiness Checklist

Before treating a local install as ready for `1.0.0` usage, confirm the paths
you depend on are documented and smoke-tested:

- Install and upgrade:
  [install.md](./install.md) explains Homebrew, GitHub Release, pinned-version,
  upgrade, PATH, Linux preview, and uninstall behavior.
- Runtime bootstrap and diagnostics:
  run `tentgent runtime bootstrap`, `tentgent doctor`, `tentgent runtime
  status`, and `tentgent --version` after install or upgrade. Runtime-home and
  diagnostic expectations are in [runtime.md](./runtime.md).
- Provider auth:
  use `tentgent auth status` and `tentgent auth mode` to confirm how provider
  keys are resolved. Secret resolution rules live in
  [auth-secrets.md](../contracts/auth-secrets.md).
- Provider compatibility:
  [provider-compatibility.md](./provider-compatibility.md) lists supported,
  partial, and unsupported OpenAI, Claude/Anthropic, and Gemini-compatible
  routes. These provider compatibility labels are separate from the stability
  tier inventory. Partial provider coverage can still have stable
  unsupported-error behavior through
  [provider-api-errors.md](../contracts/provider-api-errors.md).
- Model support status:
  use `tentgent model catalog`, `tentgent model inspect`, `tentgent server
  inspect`, and `tentgent doctor` to understand support status and proof state.
  Start with [model-support-catalog.md](./model-support-catalog.md), then use
  [model-support-status.md](../contracts/model-support-status.md) and
  [model-support-proof-schema.md](../contracts/model-support-proof-schema.md)
  for contract details.
- Daemon API usage:
  route user-facing HTTP work through [api.md](./api.md). Transport-level daemon
  guarantees live in [http-daemon.md](../contracts/http-daemon.md).
- Release smoke:
  use [model-fixtures.md](./model-fixtures.md) for small local model checks and
  [commands.md](./commands.md) for copy-paste CLI workflows.

## Contributor Readiness Checklist

Before changing behavior near the `1.0.0` boundary, keep the documentation
contract aligned with the implementation:

- Stable, experimental, internal, or deprecated surface changes update
  [api-surface-stability.md](../contracts/api-surface-stability.md).
- Runtime, storage, daemon, auth, provider, server, model support, or proof
  boundary changes update the affected files under
  [docs/contracts/](../contracts/README.md).
- Provider-compatible behavior changes update
  [provider-compatibility.md](./provider-compatibility.md),
  [provider-compatible-examples.md](./provider-compatible-examples.md), and
  [provider-api-errors.md](../contracts/provider-api-errors.md) when applicable.
- Model support status or proof behavior changes update
  [model-support-catalog.md](./model-support-catalog.md),
  [model-support-status.md](../contracts/model-support-status.md), and
  [model-support-proof-schema.md](../contracts/model-support-proof-schema.md)
  when applicable.
- Release-readiness checks run through `bash scripts/test-release-readiness.sh`
  before tagging release work.
- Documentation-only changes should still run Markdown path checks and
  `git diff --check`; behavior changes need the relevant Rust or Python test
  slices.

## Stable, Experimental, And Post-1.0

Use this section only as orientation. The detailed inventory remains in
[api-surface-stability.md](../contracts/api-surface-stability.md).

- Stable surfaces are the documented CLI commands, daemon REST routes, local
  model-bound server public routes, direct cloud server public routes, and
  provider unsupported-error semantics that the stability contract marks as
  stable.
- Experimental surfaces are documented but still allowed to tighten before or
  after `1.0.0`, especially where media workflows, runtime profile coverage,
  LoRA run execution, support proof recovery, or provider-compatible partial
  coverage is still maturing.
- Internal surfaces are implementation boundaries, including hidden commands,
  worker commands, direct Python runtime execution endpoints, daemon-local
  workspace paths, runtime process management details, and implementation-only
  proxy routes.
- Deprecated surfaces are compatibility aliases or legacy paths that remain
  callable but should not be promoted for new usage.
- Post-1.0 work includes broader provider parity, grouped serving targets, and
  the multimodal context pipeline described in
  [post-1.0-serving-targets-and-multimodal-context-pipeline.md](../plans/post-1.0-serving-targets-and-multimodal-context-pipeline.md).

## v0.9.0 Readiness Status

`v0.9.0` is the hardening track that prepares the documented `1.0.0` promise.
It covers the API surface audit, conformance smoke coverage, install and
runtime recovery diagnostics, cancellation and shutdown semantics, support
proof retry and stale-state recovery, and this readiness routing.

The `v0.9.0` planning records were archived during issue `#82`. Release tagging
remains an operator action after the closeout branch merges.
